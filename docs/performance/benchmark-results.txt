PERCOLATOR — BENCHMARK RUNBOOK & CAPTURE LOG
============================================================================================

PURPOSE
  The reference for running Percolator's benchmarks and judging the result. It answers three
  questions for whoever runs them next:
    1. WHAT to run (commands, args, runtime budget) ................ "How to run"
    2. WHAT must hold regardless of hardware (the regression gate) . "Invariants"
    3. WHAT the numbers were last time, and on what machine ........ "Capture log"

  Read INVARIANTS first. Those numbers are fixed by the data + the algorithm, NOT the CPU, so
  they must reproduce on ANY machine. If one drifts, the engine's *behaviour* changed —
  investigate before trusting anything else. THROUGHPUT, by contrast, is hardware- and
  load-dependent: only compare it against a prior run ON THE SAME MACHINE, never across
  machines, and never treat a cross-machine delta as a regression.

  Numbers here are distilled. Full console output of a run is ephemeral (e.g. /tmp at capture
  time); record the distilled result in the capture log below — append, don't overwrite. This
  file is the durable, version-controlled record. CI (GitHub Actions) also runs the benchmarks on
  every PR and uploads the raw console output as an artifact, but does NOT gate on them: throughput
  is machine-dependent, so the invariants below stay a manual comparison (see DECISIONS ADR-024).

--------------------------------------------------------------------------------------------
HOW TO RUN
--------------------------------------------------------------------------------------------
  cd engine
  export CARGO_TARGET_DIR=/tmp/perc-target            # build off-tree (see CLAUDE.md)

  # ---- correctness (the contract; fast) ----
  cargo test --release --test oracle                  # differential oracle: false_neg/pos = 0
  cargo test --release --test stress                  # 15 pressure tests (churn, par==seq, ghosts)
  cargo test --release --test stress \
      ten_million_queries_mixed_ops -- --ignored --nocapture   # 10M-scale soak (needs ~4+ GiB)

  # ---- throughput / structure ----
  cargo run --release --bin bench -- 1000000 20000 0.0      # PURE SELECTIVE (clean cand count)
  cargo run --release --bin bench -- 1000000 20000 0.05     # +broad lane (shows its ~5-9x cost)
  cargo run --release --bin segbench -- 300000 3000 0.0     # read-amp vs 1/2/4/8 segments
  cargo run --release --bin snapbench -- 1000000 200        # snapshot-publish cost (ADR-016)
  cargo run --release --bin learn -- 500000 50 0.30         # NPMI corpus learner (see appendix)

  bench args:     <num_queries> <num_titles> <broad_frac> <skew> <seed> [reps]
  segbench args:  <num_queries> <num_titles> <broad_frac> <seed>
  snapbench args: <num_queries> <iters>
  Seeds default to 0x00C0FFEE so runs are reproducible. Budget: each is <~30s except the 5M+
  bench memory passes (they build the corpus to disk twice — a few minutes) and the 10M soak.

INVARIANTS — the regression gate (machine-INDEPENDENT; must reproduce)
--------------------------------------------------------------------------------------------
  Last verified 2026-05-30 (Apple M4 Max, 16-core) AND 2026-05-27 (aarch64, 4-core): identical.

  CANDIDATES / TITLE  (the core selectivity metric — the whole engine exists to keep this small
  and FLAT as the corpus grows):
    segbench 300000 3000 0.0    K=1 -> 53.48   K=2 -> 55.11   K=4 -> 56.67   K=8 -> 57.68
    bench 1000000 20000 0.0     avg 54.5  (p95 96, p99 112)   [postings/title == cands/title]
    -> ~54, flat regardless of corpus size; postings == cands on the selective (no-broad) path.

  FILTER SKIP %  (per-segment anchor filter; rises with segment count):
    segbench K=1/2/4/8   ->   ~46% / ~63% / ~76% / ~86%      (filter heap 0.02–0.07 MB)

  COST-CLASS SPLIT  (bench 1000000 20000 0.05):   ~99.5% A | ~0.2% B | ~0.2% C | 0 D
    (broad_frac=0.0 -> 100% A.  class D must always be 0 for generated data.)

  CORRECTNESS  (must be exactly these — this is the contract):
    oracle:   false_neg = 0,  false_pos = 0      (~109k matches over 40k queries x 4k titles)
    stress:   15/15 pass;  ghosts (deleted ids in results) = 0;  par == seq, 0 mismatches
    10M soak: 0 ghosts, par==seq 0 mismatches, compaction reclaims tombstones, sources consistent

  RESIDENT MEMORY  (persistent/mmap profile — machine-independent; ADR-020), bench @1M:
    retain_source=false -> ~5–6 B/query     (sources off-heap — the production profile)
    retain_source=true  -> ~115–120 B/query (the source store, ~113 B/q, dominates)
    components: dict ~5 B/q (bounded by vocab), alive overlay ~1 B/q, logical_index ~0 (mmap'd)

--------------------------------------------------------------------------------------------
MACHINE-DEPENDENT — record the machine; compare only same-machine over time
--------------------------------------------------------------------------------------------
  Throughput (titles/sec/core selective & broad, build q/s, updates/s, snapshot publish/s,
  parallel speedup) scales with the CPU, memory bandwidth, and current machine load. Do NOT
  compare across machines or call a delta a regression without re-running the prior machine.
  Parallel speedup saturates on memory bandwidth (and on big.LITTLE CPUs the efficiency cores
  drag the rayon reduce), so expect sub-linear scaling past ~4 threads — the 4-core reference's
  "~3.8x on 4 threads" does NOT extrapolate to a 16-core box.

  macOS caveat: bench's "process RSS" reads /proc/self/status (Linux-only) and prints 0 on
  macOS. Ignore it; use the persistent/mmap RESIDENT breakdown above, which is computed from
  Engine::metrics() and is machine-independent.
